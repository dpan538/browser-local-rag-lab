# V4.1 Cross-Model Validation Execution Plan

Generated: 2026-06-10

## Purpose

V4.1 tests whether the Round 03 reliability architecture is model-agnostic. The target question is not whether another model is faster than Qwen. The target question is whether the same deterministic lanes, post-generation prose polisher, evidence tag injection, contract gate, and quality gates continue to work when the prose body is produced by a non-Qwen model.

## Scientific Claim Under Test

If a different browser-local or local small model supplies only the prose body, the surrounding system should still provide:

- deterministic refusal behavior;
- deterministic source/rights output;
- exact evidence tag visibility;
- postprocessed caution against overconfidence and unsupported first/earliest claims;
- the same generation contract gate.

This would support the claim that the reliability layer is independent of any single model family.

## Current V4.1 Artifacts

- Universal 50-query packet: `reports/v41_universal_packets_pilot50.json`
- Universal 300-query packet: `reports/v41_universal_packets_300.json`

The universal packet contains the model prompt, expected deterministic answers, expected evidence tags, required fields, and evidence records. It is designed to be runtime neutral.

## Pilot50 Procedure

1. Generate or refresh the universal packet and raw-generation template.

```bash
npm run v41:packets:pilot50
npm run v41:raw-template:pilot50
```

2. Run a non-Qwen model against the `model_prompt` column in `reports/v41_raw_generation_template_pilot50.jsonl`.

The model runner must fill a JSONL file at:

```text
reports/v41_raw_generations_pilot50.jsonl
```

Each row should contain:

```json
{
  "query_id": "BQ01",
  "generated_text": "Based on these records, ...",
  "runtime_model_id": "model/name",
  "runtime_name": "transformers_js_or_other_runtime",
  "generation_status": "completed",
  "ttft_ms": 0,
  "total_latency_ms": 0,
  "output_tokens": 0,
  "tokens_per_second": 0,
  "cache_state": "warm_from_previous"
}
```

Only model-generation rows are required. Deterministic refusal and source/rights rows are generated by the system finalizer.

3. Finalize, import, and contract-check the run.

```bash
npm run v41:finalize:pilot50
```

This command writes:

- `reports/v41_cross_model_pilot50_export.json`
- `reports/v41_cross_model_pilot50.json`
- `reports/v41_cross_model_pilot50_answers.jsonl`
- `reports/V41_CROSS_MODEL_PILOT50.md`

4. Run the quality gates using the finalized answer JSONL.

Recommended commands, with paths adjusted after the first pilot:

```bash
node scripts/detect_hallucination.mjs \
  --labels fixtures/expansion/round03_300/labels.jsonl \
  --records fixtures/expansion/round03_300/records.jsonl \
  --queries fixtures/expansion/round03_300/queries.jsonl \
  --answers reports/v41_cross_model_pilot50_answers.jsonl \
  --json-out reports/hallucination_v41_pilot50.json \
  --md-out reports/HALLUCINATION_V41_PILOT50.md \
  --strict

node scripts/detect_misreading.mjs \
  --queries fixtures/expansion/round03_300/queries.jsonl \
  --labels fixtures/expansion/round03_300/labels.jsonl \
  --records fixtures/expansion/round03_300/records.jsonl \
  --answers reports/v41_cross_model_pilot50_answers.jsonl \
  --json-out reports/misreading_v41_pilot50.json \
  --md-out reports/MISREADING_V41_PILOT50.md \
  --strict

node scripts/check_guardrail_compliance.mjs \
  --labels fixtures/expansion/round03_300/labels.jsonl \
  --records fixtures/expansion/round03_300/records.jsonl \
  --answers reports/v41_cross_model_pilot50_answers.jsonl \
  --json-out reports/guardrail_compliance_v41_pilot50.json \
  --md-out reports/GUARDRAIL_COMPLIANCE_V41_PILOT50.md \
  --strict
```

## Pilot50 Success Criteria

The V4.1 pilot can advance beyond 50 rows only if:

| Gate | Required result |
|---|---|
| Runtime completion | 50/50 completed |
| Runtime error rows | 0 |
| Contract findings | 0 fail / 0 warn |
| Deterministic lanes | 0 fail and near-zero latency |
| Hallucination gate | no strict failures |
| Misreading gate | no strict failures |
| Guardrail compliance | no strict failures |

Latency does not need to beat Qwen for the pilot to count as successful. The primary claim is model-agnostic reliability.

## Full 300 Expansion Criteria

Run the 300-query V4.1 only after pilot50 passes. Use:

```bash
npm run v41:raw-template:300
```

Then fill:

```text
reports/v41_raw_generations_300.jsonl
```

and run:

```bash
npm run v41:finalize:300
```

The 300-query result should be compared to V3.3 on:

- contract fail/warn count;
- deterministic row count and deterministic latency;
- model-generation average latency and P95 latency;
- hallucination/misreading/guardrail findings;
- prose usability.

## Interpretation Rules

- Deterministic lane performance must be reported as system latency, not model generation latency.
- Contract pass is expected to be model-independent because evidence tags and source/rights fields are system-generated.
- Differences between models should be discussed mainly in prose quality and model-generation latency, not in field-level contract compliance.
- If the alternate model fails quality gates, that does not invalidate the architecture; it marks the model prose body as weaker while preserving the system-level contract result.
